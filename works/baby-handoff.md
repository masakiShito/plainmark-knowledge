# baby-handoff — 夫婦の育児をつなぐ引き継ぎWebアプリ

## 概要

baby-handoffは、夫婦で育児ログと引き継ぎ情報を共有するためのスマートフォン向けWebアプリです。

育児記録を細かく残すこと自体ではなく、交代した側が「今日は何があったか」「次に何をすればよいか」を短時間で把握できることを重視しています。

ごはん、睡眠、排泄、お風呂、体温、薬、保湿などの記録をタイムラインで確認でき、別枠で引き継ぎメモも残せます。家族単位でデータを分離し、認証した利用者が自分の家族データだけを扱えるように設計しました。

- GitHub: https://github.com/masakiShito/baby-handoff
- ステータス: MVP実装済み、家族招待などを開発中

## 課題

育児中は、口頭での引き継ぎだけでは次のような情報が抜けやすくなります。

- 最後に食事をした時刻
- 昼寝の時間
- 薬や保湿を実施したか
- 体調面で気になったこと
- 次に対応してほしいこと

一般的な育児記録アプリは記録項目が豊富な一方で、交代する家族が必要な情報をすぐ確認する用途では情報量が多くなりがちです。

そこで、詳細な成長記録ではなく「今日の状態」と「引き継ぎ」に焦点を当て、夫婦間の認識差を減らすアプリとして企画しました。

## 担当範囲

個人開発として、以下を担当しています。

- 課題整理とプロダクトコンセプトの策定
- 画面構成とスマートフォン向けUI設計
- Next.js App Routerを使ったフロントエンド実装
- Supabase Authを使った認証
- 家族・子ども・育児ログ・引き継ぎメモのDB設計
- Server Actionsによる登録・更新処理
- Row Level Securityによるアクセス制御
- Zodによる入力値検証
- DockerとSupabase CLIを使ったローカル開発環境の構築

## 技術構成

### Next.js / TypeScript

App RouterとServer Actionsを採用し、認証、初期セットアップ、育児ログ登録、引き継ぎメモ登録を機能単位で整理しています。

DBのレコード形式を画面で扱う型へ変換するmapperを用意し、データ取得処理とUIの責務を分けました。

### Supabase

認証、PostgreSQL、Row Level Securityをまとめて利用しています。

家族スペースをデータ分離の単位とし、RLSで自分が所属する家族の情報だけを取得・更新できるようにしました。Server Actions側でも認証状態と権限を確認し、ブラウザ側の制御だけに依存しない構成です。

### Tailwind CSS

スマートフォンで片手操作しやすいカード中心のUIを構築しています。

「Notion × LINE × 母子手帳」をデザインの方向性とし、やさしさを保ちつつ、育児中でも情報を見失いにくい余白と文字サイズを意識しました。

### Docker / Supabase CLI

Next.jsとSupabaseのローカル環境を再現できるようにし、DBマイグレーションをリポジトリ内で管理しています。

## 主な機能

- メールアドレスとパスワードによるサインアップ・ログイン
- 家族スペースと子どもの初期登録
- 今日の育児状況をまとめたサマリー
- 育児ログのタイムライン表示
- ごはん、睡眠、うんち、お風呂、体温、薬、保湿、メモの登録
- 引き継ぎメモの登録・表示
- 家族単位のデータアクセス制御
- スマートフォン向けレスポンシブUI

## 工夫した点

### 「記録」ではなく「引き継ぎ」を中心にした

何でも記録できるアプリにすると入力負担が増えるため、交代する家族が知りたい情報に焦点を絞りました。

今日の状態、時系列のログ、明示的な引き継ぎメモを同じダッシュボードで確認できるようにしています。

### セキュリティを画面制御だけで済ませない

家族データはプライベートな情報のため、UIで非表示にするだけではなく、SupabaseのRLSでDBアクセス自体を制限しました。

さらにServer Actionsでも認証・権限チェックを行い、複数層でアクセスを確認しています。

### 忙しい状況での操作を想定する

育児中は片手操作や短時間の確認が多いため、最大幅を抑えたモバイルファースト設計にしました。

入力項目を増やしすぎず、カード、余白、大きめの角丸を使って、状態と操作対象を見分けやすくしています。

### 将来機能を追加しやすいデータ構造

AI要約、週次レポート、通知などを将来追加できるように、育児ログと引き継ぎメモを構造化して保存しています。

## 学び

生活上の課題を解決するアプリでは、機能数よりも、利用する場面を具体的に想定することが重要だと学びました。

また、Supabaseを利用することで開発速度を上げられる一方、RLSの条件、Server Actionsの認証、ブラウザ用とサーバー用クライアントの使い分けを明確にする必要がありました。

UIについても、情報を多く見せることより、忙しい利用者が迷わない順序で配置することを優先しました。
