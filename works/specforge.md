# SpecForge — 設計書の品質と再現性を高めるドキュメント基盤

## 概要

SpecForgeは、設計書を単なるMarkdownや文章ファイルではなく、検証可能な構造化データとして扱うためのプラットフォームです。

設計書の品質が執筆者の経験や記述スタイルに依存すると、必要な項目が不足したり、名称が揺れたり、別資料への参照が切れたりします。見た目が整っていても、設計として必要な情報がそろっているとは限りません。

SpecForgeでは、設計書の内部モデル、入力UI、品質ルール、整合性検査、出力処理を分け、誰が書いても一定の品質を保ちやすい仕組みを目指しています。

現在は、プロダクト方針、アーキテクチャ、モノレポ構成、Docker開発環境などの基盤を整備し、中核機能を段階的に実装しています。

- GitHub: https://github.com/masakiShito/specforge
- ステータス: アーキテクチャ・開発基盤を構築済み、中核機能を開発中

## 課題

設計書の運用では、次のような問題が起こりやすくなります。

- 執筆者によって章立てや粒度が異なる
- 必須項目が抜けてもレビューまで気づけない
- 用語や命名が資料ごとに揺れる
- 画面、API、DBなどの参照関係が切れる
- 過去の設計を再利用しにくい
- Markdownの見た目は整っていても、内容の妥当性を機械的に確認できない

テンプレートを配布するだけでは、項目を削除したり自由記述へ戻したりできるため、品質を安定させるには限界があります。

そこで、設計書を最初から構造化されたモデルとして保持し、入力時と保存時にルールを適用する仕組みを考えました。

## 技術構成

### Next.js / TypeScript

構造化フォームによる設計書編集、検証結果、品質警告、プレビューなどを提供するWeb UIとして利用します。

画面固有の実装と共通UIを分け、設計書の項目定義はUIへ直接埋め込まず、スキーマから扱える構成を目指しています。

### FastAPI / Python

設計書の保存、検証、出力、将来のAI補助提案を担当するAPIとして利用します。

UIだけで検証を完結させず、API側でも同じ品質ルールを適用できる構成を想定しています。

### PostgreSQL

設計書本文を単一の文字列として保存するだけではなく、ドキュメント、セクション、フィールド、参照関係、検証結果などを扱えるデータ基盤として採用しています。

### pnpm Monorepo

アプリケーションと共通パッケージを次の責務に分けています。

- `apps/web`: 設計書編集UI
- `apps/api`: 保存・検証API
- `packages/ui`: 共通UIコンポーネント
- `packages/document-schema`: 設計書構造と制約
- `packages/editor-engine`: 出力形式に依存しない内部モデル
- `packages/lint-rules`: 設計書品質ルール

### Docker Compose

Web、API、PostgreSQLをまとめて起動できるようにし、ローカル環境の差を減らしています。

## 目指している主な機能

## 工夫した点

### Markdownを内部モデルにしない

Markdownは扱いやすい出力形式ですが、自由度が高いため、設計項目の存在や参照関係を保証しにくいという問題があります。

SpecForgeでは、設計内容を構造化モデルとして保持し、Markdownは複数ある出力形式の一つとして扱います。

### 品質ルールをエディタから分離する

検証処理を画面コンポーネントへ直接書くと、APIやバッチ処理から再利用できません。

そこで、品質ルールを`lint-rules`パッケージへ分離し、入力時・保存時・CIなど複数の場所で同じ基準を利用できる構成を目指しています。

### 汎用化しすぎず責務を明示する

何でも入る`utils`や`common`を増やすのではなく、document-schema、editor-engine、lint-rulesのように目的が分かる単位でパッケージを分けています。

### 実装前に思想と境界を文書化する

プロダクトビジョン、品質原則、リポジトリ構成、アーキテクチャ原則をドキュメントとして残しています。

機能追加のたびに構成が変わるのではなく、判断基準を先に共有できる状態を重視しました。

## 学び

ドキュメントツールを設計する際は、エディタの使いやすさだけでなく、何を正しい設計書とみなすかという品質モデルが必要だと学びました。

また、出力形式と内部モデルを分離すると初期設計は複雑になりますが、Markdown以外の形式への展開や、機械的な検証を行いやすくなります。

モノレポでは、コードを一つの場所へ集めるだけでなく、各パッケージの依存方向と責務を明確にしなければ、かえって変更範囲が分かりにくくなることも意識しました。
