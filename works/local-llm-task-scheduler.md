# Local LLM Task Scheduler — 再現性を重視した予定生成システム

## 概要

Local LLM Task Schedulerは、Todoと工数・期限・優先度を持つ計画タスクを統合し、その日の空き時間へ実行可能な時間割を割り当てるタスク管理システムです。

特徴は、スケジューリングそのものをLLMへ任せず、決定論的なルールベース処理で行う点です。LLMは、生成した計画の説明や、予定に入りきらなかったタスクの理由を自然言語で伝える補助役として利用します。

同じ入力から同じ結果を再現できること、計画が破綻した場合に理由を説明できること、最終判断をユーザーへ残すことを重視して設計しています。

現在は要件定義、基本設計、詳細設計まで完了し、実装へ移行する段階です。

- GitHub: https://github.com/masakiShito/local-llm-task-scheduler
- ステータス: 設計完了、実装準備中

## 課題

生成AIへ「今日の予定を組んで」と依頼すると、それらしい計画は作れますが、次の問題があります。

- 同じ条件でも結果が変わる
- なぜその順番になったか検証しにくい
- 所要時間や固定予定との整合性が崩れることがある
- 実行不能な計画でも自然な文章で提示される
- テストによって正しさを保証しにくい

一方、単純なTodoアプリでは、優先度や期限は管理できても、実際の空き時間へ割り当てるところまでは支援しないものが多くあります。

そこで、時間割の生成はテスト可能なアルゴリズムで行い、LLMは説明に限定する構成を考えました。

## 技術構成

### Next.js

タスク入力、固定予定登録、生成された時間割、警告、overflowを表示するフロントエンドとして利用します。

生成結果を自動確定するのではなく、ユーザーが確認・判断できる提案画面を中心に設計しています。

### FastAPI

タスク管理、固定予定管理、スケジュール生成、計画保存、LLM説明生成をAPIとして提供します。

スケジューリングロジックとLLM呼び出しを別の責務に分け、AIサービスが利用できない場合でも計画生成自体は行える構成です。

### PostgreSQL

Todo、計画タスク、固定予定、生成したPlan、PlanBlockなどを保存します。

生成結果も永続化し、入力条件と出力を後から確認できるようにすることで、再現性と監査性を確保します。

### Ollama / Local LLM

外部AI APIへ個人の予定やタスクを送らず、ローカル環境で説明文を生成する想定です。

LLMの出力は自由文のまま扱わず、JSON Schemaで形式を固定し、アプリ側で検証してから表示します。

## 設計した主な機能

- Todoと計画タスクの登録・更新・削除
- 工数、期限、優先度、分割可否の管理

## 工夫した点

### AIに最適化を丸投げしない

時間計算や優先順位付けは、Greedy法と明示的なルールで処理します。

これにより、スケジュール結果を単体テストでき、同じ入力に対して同じ出力を期待できます。

### LLMを交換可能な補助機能にする

LLMは説明生成だけを担当し、スケジューリングの成否には影響しない構成にしています。

モデル変更やLLM停止があっても、主要機能が利用不能にならないことを重視しました。

### 「予定に入らない」を正常な結果として扱う

すべてのタスクを無理に詰め込まず、入らなかったタスクをoverflowとして返します。

期限超過、空き時間不足、分割不可など、入らなかった理由も構造化して保持し、ユーザーが次の判断を行えるようにします。

### 実装前に境界条件を文書化する

画面、API、バリデーション、エラー、永続化、トランザクション、テストまで分けて設計書を作成しました。

AIを含む機能ほど曖昧なまま実装を始めず、決定論的な部分と非決定論的な部分の境界を先に決めることを意識しています。

## 学び

AI機能を導入する際は、何をAIへ任せるかだけでなく、何を任せないかを決めることが重要だと学びました。

特にスケジューリングのように時間や制約の整合性が必要な処理では、LLMの柔軟さよりも、アルゴリズムの再現性とテスト可能性が適しています。

また、要件定義からテスト設計までを通して整理することで、APIやDBの実装前に矛盾や不足を発見しやすくなりました。
