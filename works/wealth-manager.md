# Wealth Manager — 老後資産を見える化するシミュレーション基盤

## 概要

Wealth Managerは、現在の資産・毎月の収支・積立状況・老後の生活条件を整理し、将来の資産が足りるかを確認するためのWebアプリです。

資産額だけを入力して結果を出す簡易計算機ではなく、口座、資産種別、月次収支、積立計画、老後条件などを継続的に管理できるアプリケーションを目指しています。

現在は、Dockerによる開発環境、JWT認証、DBマイグレーション、資産種別・口座・月次収支のCRUD APIまで実装しています。老後資産シミュレーションとフロントエンド画面は今後実装予定です。

- GitHub: https://github.com/masakiShito/wealth-manager
- ステータス: バックエンド基盤を実装済み、主要画面と計算機能を開発中

## 課題

老後資産を考える際は、単に「何千万円必要か」という一般的な数字だけでは、自分の状況に合った判断ができません。

実際には、次の条件を組み合わせて考える必要があります。

- 現在の現金・預金・投資資産
- 毎月の収入と支出
- 積立可能額
- 退職までの年数
- 想定利回り
- 退職後の生活費
- 年金収入
- 何歳まで資産を使う想定か

これらを表計算ソフトで管理することもできますが、入力項目や計算式が分散しやすく、条件を変更した比較も行いにくくなります。

そこで、日常の資産管理と将来シミュレーションを同じデータモデルで扱うWebアプリとして設計しました。

## 技術構成

### Next.js / TypeScript

フロントエンドにはNext.jsのApp Routerを採用しています。

画面、共通コンポーネント、機能単位のモジュール、API通信、型定義を分け、今後ダッシュボードやシミュレーション画面が増えても責務を追いやすい構成を目指しています。

### FastAPI / Python

バックエンドはFastAPIで構築し、APIを`v1`配下にまとめています。

エンドポイント、サービス層、DBモデル、入出力スキーマを分離し、計算ロジックを画面やDBアクセスから独立させられる構成にしています。

### MySQL / SQLAlchemy / Alembic

資産種別、口座、月次収支などをリレーショナルデータとして管理します。

SQLAlchemyでモデルを定義し、Alembicでスキーマ変更を履歴管理しています。コンテナ起動時にマイグレーションを適用できるため、環境ごとの差異を減らせます。

### Docker Compose

フロントエンド、バックエンド、MySQLを別コンテナとして起動します。

ローカル環境を手作業で構築せず、同じ手順で開発を始められることを重視しました。

## 工夫した点

### シミュレーションより先にデータ基盤を作る

将来資産の計算式だけを先に作るのではなく、現在の資産と収支を継続的に保存できる構造から実装しています。

これにより、将来的に積立実績と計画の差分や、時系列での資産推移も扱えるようにしました。

### APIの責務を分離する

ルーティング、入出力スキーマ、ビジネスロジック、DBアクセスを分けています。

資産シミュレーションの条件が増えても、エンドポイント内に計算処理が集中しないようにすることを意識しました。

### DB変更を再現可能にする

開発途中でテーブル構造が変わることを前提に、初期段階からAlembicを導入しました。

SQLファイルを手作業で適用するのではなく、マイグレーションとして変更履歴を残せるようにしています。

### 金融アプリとして誤解を招かない構成にする

このアプリは投資商品の推奨や将来利益を保証するものではなく、ユーザーが入力した前提条件を整理・比較するためのシミュレーションツールとして設計しています。

## 学び

複数の条件を扱うシミュレーションアプリでは、計算式だけでなく、前提条件をどの単位で保存し、後から比較できるようにするかが重要だと学びました。

また、FastAPIとNext.jsを分離する構成では、認証方式、APIのバージョニング、エラー形式、型の対応を早い段階で決める必要があります。

フロントエンドが未完成でも、SwaggerとHealth Checkを用意することで、バックエンド単体の動作確認と開発を進めやすくなりました。
